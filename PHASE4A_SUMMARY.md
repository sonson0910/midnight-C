## What Each Type Does

### 1. **ProofData** - The Raw Proof
```cpp
struct ProofData {
    std::vector<uint8_t> proof_bytes;  // 128-4096 bytes typically
};
```
**Generated by**: Proof Server
**Used for**: Blockchain verification
**Privacy**: Proves computation without revealing witness

### 2. **PublicInputs** - The Disclosed Values
```cpp
struct PublicInputs {
    std::map<std::string, std::string> inputs;  // field_name => hex_value
};
```
**Example Values**:
- `state: "0000...0000"` (board vacant)
- `owner: "abc...def"` (owner commitment)
- `message: "48656c6c6f"` ("hello" in hex)

**On-Chain**: These are visible to everyone
**Private**: The secret key that generated `owner` stays hidden

### 3. **WitnessOutput** - The Private Witness Data
```cpp
struct WitnessOutput {
    std::vector<uint8_t> output_bytes;  // 32 bytes = secret key
    std::string witness_name;           // "localSecretKey"
};
```
**Created by**: TypeScript witness function
**Sent to**: Proof Server
**Never**: Exposed on-chain

**Example Flow**:
```
TypeScript (localSecretKey):
  returns → [secret_key_bytes, state]

Proof Server (generate_proof):
  receives → secret_key_bytes + public_inputs
  generates → ZK proof that proves:
    "I know the secret key that generated this commitment"
    WITHOUT revealing the secret key
```

### 4. **CircuitProof** - The Complete Package
```cpp
struct CircuitProof {
    ProofData proof;                    // The actual ZK proof
    PublicInputs public_inputs;         // Disclosed values
    CircuitProofMetadata metadata;      // Circuit info
    uint64_t generated_at_timestamp;    // When generated
    CircuitProof::VerificationStatus verification_status;  // UNVERIFIED → VERIFIED
};
```
**Flow**:
```
CircuitProof CREATED (proofgen)
    ↓
CircuitProof LOCALLY_VERIFIED (size + inputs valid)
    ↓
CircuitProof SERVER_VERIFIED (Proof Server confirms)
    ↓
CircuitProof BLOCKCHAIN_VERIFIED (RPC confirms on-chain)
```

### 5. **ProofEnabledTransaction** - Ready for Blockchain
```cpp
struct ProofEnabledTransaction {
    std::string transaction_hex;           // UTXO transaction
    CircuitProof circuit_proof;            // The ZK proof
    std::string ed25519_signature;         // From Phase 3
    std::map<std::string, std::string> ledger_updates;  // State changes
};
```
**Ready for**: Direct RPC submission to network

## The Proof Server Integration

**Phase 4A provides the client interface to Proof Server:**

```cpp
ProofServerClient client;
if (client.connect()) {
    // Send circuit data + inputs + witnesses
    ProofResult result = client.generate_proof(
        "post",                  // Circuit name
        circuit_data,           // Compiled circuit binary
        public_inputs,          // Disclosed values
        witnesses               // Private witness outputs
    );

    if (result.success) {
        // Proof is ready for blockchain
        proof_enabled_tx.circuit_proof = result.proof;
    }
}
```

## Real Example: Bulletin Board

### Circuit (Compact Language)
```compact
export circuit post(newMessage: string): [] {
    assert(state == State.VACANT);
    owner = disclose(publicKey(localSecretKey(), sequence));
    message = disclose(some(newMessage));
    state = State.OCCUPIED;
}

witness localSecretKey(): Bytes<32>;  // Private!
```

### TypeScript Witness Function
```typescript
export const witnesses = {
    localSecretKey({ privateState }): [State, Uint8Array] {
        return [privateState, privateState.secretKey];
    }
};
```

### C++ (Phase 4A) Handles This
```cpp
// Receive witness output from TypeScript
WitnessOutput witness;
witness.witness_name = "localSecretKey";
witness.output_bytes = secret_key_32_bytes;

// Create public inputs
PublicInputs inputs;
inputs.add_input("state", "0");  // VACANT
inputs.add_input("message", "48656c6c6f");  // "hello" hex

// Send to Proof Server
ProofServerClient client;
ProofResult result = client.generate_proof(
    "post",
    circuit_data,
    inputs,
    { {"localSecretKey", witness} }
);

// Result contains ZK proof that proves:
// "This message was posted while preserving poster anonymity"
//  (without revealing the actual secret key)
```

## Key Design Aspects

### 1. Full Serialization Support
**Why**: Network transmission requires standard formats
**How**: Every type implements:
```cpp
json to_json() const;
static Type from_json(const json& j);
```

### 2. Type Safety
**Why**: Cryptographic data requires strict handling
**How**: Strong C++ types instead of raw JSON everywhere

### 3. Proof Lifecycle Tracking
**Why**: Need to know verification status
**How**: `CircuitProof::VerificationStatus` enum tracks:
- UNVERIFIED (just created)
- LOCALLY_VERIFIED (size/format OK)
- SERVER_VERIFIED (Proof Server confirmed)
- BLOCKCHAIN_VERIFIED (RPC confirmed)

### 4. Extensible Fields
**Why**: Different circuits need different public inputs
**How**: Map-based public inputs support any fields
